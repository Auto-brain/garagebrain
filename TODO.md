## 2. Раздельная стоимость в обслуживании: работа + материалы

**Суть:** в записи обслуживания может быть отдельно **стоимость работ** и
**стоимость материалов/запчастей**. Стоимость материалов **необязательна**
(может входить в стоимость работ), а стоимость работ может быть **0**
(«заменил сам»).

**Примеры:**
- Замена масла: работа **10 BYN**, масло **150 BYN**.
- `Замена воздушного фильтра, 165666, 0, 20 BYN` → работа `0` (сделал сам),
  материалы `20 BYN`.

### Формат ввода (предложение)
`Название работ, пробег, стоимость работ, [стоимость материалов] [валюта]`

- Стоимость материалов — опциональна.
- Стоимость работ `0` = сделал сам.

### Что нужно
- [ ] Разделить стоимость записи на **`labor_cost`** (работа) и **`parts_cost`**
  (материалы/запчасти).
- [ ] `parts_cost` — опционально (NULL, если не указано / входит в работу).
- [ ] `labor_cost = 0` — валидный случай («сделал сам»).
- [ ] Итоговая стоимость записи = `labor_cost + parts_cost` (для статистики).
- [ ] Парсер чата распознаёт **два числа** (работа, материалы) + валюту.
- [ ] Статистика: разбивка расходов **работа vs материалы**.
- [ ] **Топливо как отдельная сумма затрат.** Для заправки: работа = 0, стоимость
  топлива пишется в `parts_cost` (строка «Стоимость топлива») → уже попадает в
  категорию «Топливо» статистики. Доработать: показать **отдельную метрику
  «Затраты на топливо»** на дашборде и связать со `fuel_records` (литры → л/100км +
  суммарный расход денег), чтобы не было двух независимых источников затрат на ДТ.

### Где править (ориентир)
- `shared/db/garagebrain_schema.sql` — `service_records`: `labor_cost INT`,
  `parts_cost INT` (оставить `cost` как сумму или вычислять).
- `internal/model/record.go` — `LaborCost *int`, `PartsCost *int`.
- `internal/service/parser.go` — парсинг формата с двумя суммами; поля в
  `---ЗАПИСЬ---` (`Стоимость работ`, `Стоимость материалов`).
- `internal/prompt/system.go` — обновить формат записи для AI.
- `internal/handler/stats.go` — разбивка по работам/материалам.
- Frontend: `RecordCard` — показывать работу и материалы отдельно;
  `ExpenseChart` — категория «работа/материалы».

> ⚠️ Поля стоимости трогают и этот пункт, и п.4 (валюта). Лучше спроектировать
> схему `service_records` (labor/parts + currency) **за одну миграцию**, даже если
> валютную логику включите позже.

## 4. Страна, валюта отображения и конверсия валют

**Суть:** пользователь выбирает страну и валюту, в которой хочет видеть стоимость.
При этом стоимость в записи может быть указана в **другой** валюте — её нужно
конвертировать в валюту отображения.

**Пример:** клиент выбрал отображение в **рублях РФ**, но стоимость указал в
**долларах** или **евро** → система пересчитывает и показывает в рублях.

> ✅ Выбор **страны/региона** в профиле уже реализован — здесь остаётся валюта и
> конверсия.

### Страна/регион на уровне авто (переопределение аккаунта)
- [ ] Страна/регион берутся из аккаунта по умолчанию, но **могут быть заданы
  отдельно для каждого авто** (напр. одна машина в Беларуси, другая в России).
- [ ] **Фолбэк:** если у авто страна/регион не указаны — использовать значения
  аккаунта. (Резолв: `car.country ?? user.country`, `car.region ?? user.region`.)
- [ ] Влияет на цены топлива по региону (п.3) и на валюту по умолчанию для записей
  этого авто (п.4): валюта/цены резолвятся по эффективному региону авто.
- [ ] `cars`: добавить **nullable** `country`, `region` (NULL = «как у аккаунта»).
- [ ] UI: в карточке/настройках авто — необязательные поля страны/региона.

### Что нужно
- [ ] **Валюта по умолчанию в настройках** — пользователь задаёт её один раз в
  профиле/настройках; она подставляется автоматически, если в записи валюта не
  указана явно (напр. «3500» без символа = валюта по умолчанию). Можно
  предзаполнять по выбранной стране, но оставлять возможность сменить вручную.
- [ ] Выбор **валюты отображения** (RUB, USD, EUR, BYN, UAH, … — список).
- [ ] Возможность вводить стоимость в **любой** валюте (в чате и в форме), напр.
  `150 BYN`, `$50`, `20€`, `3000 ₽`.
- [ ] **Конверсия** введённой суммы в валюту отображения (курсы — из п.3).
- [ ] **Все записи пересчитывать в валюту по умолчанию** клиента — и для
  отображения, и для агрегатов (итоги, статистика, графики считаются в одной
  валюте, чтобы суммы были корректны).
- [ ] **Если в записи валюта не указана** — считать, что сумма уже в валюте по
  умолчанию клиента (не конвертировать). Конверсия нужна только когда валюта
  записи отличается от валюты по умолчанию.
- [ ] Желательно конвертировать **по курсу на дату записи**, а не на текущую.

### Как хранить (предложение)
- Хранить **исходную сумму + код валюты ввода** (`amount`, `currency`), а
  отображать пересчёт в `display_currency` пользователя — чтобы не терять
  оригинал и не зависеть от изменения курса.
- `users`: добавить `display_currency` (страна/регион уже есть).
- `cars`: добавить nullable `country`, `region` (переопределение аккаунта; NULL =
  брать из аккаунта).
- `service_records` / `fuel_records`: добавить `currency` (код валюты суммы).

### Где править (ориентир)
- `shared/db/garagebrain_schema.sql` — поля `display_currency`, `currency`.
- `internal/model/{user,record}.go` — новые поля.
- `internal/service/parser.go` — распознавание валюты из текста (`BYN`, `$`, `€`, `₽`).
- `internal/service/currency.go` — конверсия по кэшу курсов (из п.3).
- `internal/prompt/system.go` — научить AI извлекать валюту.
- Frontend: селектор валюты в настройках; форматирование сумм в выбранной
  валюте; показ исходной валюты в карточке записи.

## 6. Совместный доступ к авто несколькими аккаунтами + монетизация

**Суть:** сейчас авто принадлежит **одному** `user_id`. Нужно, чтобы одним авто
могли пользоваться **несколько аккаунтов** — например, супруги, или
**владелец + арендатор**. В будущем — основа для **монетизации** (платный тариф:
делиться авто с N людьми, сдавать в аренду на время и т.п.).

> Строится поверх п.5/Варианта A: связывание аккаунтов и `mergeUsersTx` (перенос
> `cars`/записей) уже есть — здесь добавляется **много-ко-многим** доступ.

### Модель данных
- [ ] Таблица **`car_members (car_id, user_id, role, invited_by, created_at, expires_at)`**
  — много пользователей на одно авто.
- [ ] **Роли**: `owner` (полные права, удаление/приглашения), `driver`/`co-owner`
  (добавлять записи, заправки, напоминания), `renter` (доступ на срок —
  `expires_at`), `viewer` (только чтение).
- [ ] Миграция: для каждого существующего `cars.user_id` создать запись
  `car_members(role='owner')` (бэкфилл).

### Доступ и логика
- [ ] Заменить проверку владельца `cars.user_id == userID` (`handler/authz.go`)
  на проверку **членства** в `car_members` (+ учёт роли для операций).
- [ ] `GET /api/cars` и бот (`GetUserCars`/`GetActiveCar`) — отдавать авто, где
  пользователь **участник**, а не только владелец.
- [ ] Матрица прав: запись/заправку — `owner`/`driver`/`renter`; удаление авто и
  управление участниками — только `owner`; `viewer` — только чтение.
- [ ] У `renter` доступ автоматически истекает по `expires_at`.

### Приглашения (переиспользовать паттерн deep-link/кода из п.5/A)
- [ ] Владелец создаёт инвайт на авто → одноразовый токен/код (можно расширить
  `account_link_tokens` полем `car_id` + `role`, либо отдельная таблица).
- [ ] Приглашённый принимает (через веб или бот `/start join_<token>`) → попадает
  в `car_members` с заданной ролью.

### Монетизация (на будущее)
- [ ] Free: авто можно делить максимум с 1 аккаунтом; Pro: без лимита.
- [ ] Роль `renter` с `expires_at` — основа для сценария аренды (в т.ч. платной).
- [ ] Лимиты проверять при создании инвайта.

### Где править (ориентир)
- `shared/db/garagebrain_schema.sql` — `car_members` + бэкфилл.
- backend: `handler/authz.go` (членство+роль), `db/cars.go` (списки через join),
  `handler/members.go` (инвайт/принять/список участников/удалить/сменить роль).
- gateway: `db/queries.go` (`GetUserCars`/`GetActiveCar` через `car_members`),
  processor — `/start join_<token>` и команда «поделиться».
- Frontend: экран «Участники авто» (пригласить по ссылке, список, роли, удалить).
